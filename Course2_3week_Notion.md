# < Advie for applying machine learning >

## 1. Intro
### 1). 머신러닝 개발 시 흔히 겪는 문제 상황

주택 가격 예측을 위해 **정규화된 선형 회귀(Regularized Linear Regression)** 모델을 만들어 학습시켰는데, **예측 오차가 너무 크게 발생**했다고 가정해 봅시다.

이때 성능을 올리기 위해 시도할 수 있는 선택지는 수없이 많습니다.

```
                  ┌──> [데이터 추가]  훈련 데이터(Training Examples) 더 수집하기
                  ├──> [특징 축소]  너무 많은 Feature 중 일부 제거하기
                  ├──> [특징 추가]  주택의 다른 속성(Feature) 찾아 새로 추가하기
[예측 오차가 큼!] ┼──> [다항식 특징] x1^2, x2^2 등 다항식(Polynomial) 특징 추가하기
                  ├──> [lambda 감소] 정규화 파라미터 λ 줄이기
                  └──> [lambda 증가] 정규화 파라미터 λ 늘리기
```

### 2). 머신러닝 프로젝트의 성패를 가르는 핵심: "진단(Diagnostics)"

실무에서 미숙한 팀은 단순히 **"데이터가 더 많으면 성능이 좋아지겠지"** 하고 며칠, 혹은 **몇 달 동안 데이터 수집에만 시간을 투자**하곤 합니다. 하지만 데이터 추가 수집이 성능 향상에 **아무런 도움을 주지 못하는 상황**도 매우 흔합니다.

시간을 허비하지 않기 위해서는 진단(Diagnostic)을 수행해야 합니다.

- **진단(Diagnostic)이란?**
    
    - 모델이 왜 잘 작동하지 않는지, 원인이 무엇인지 파악하기 위해 실행하는 **검증 테스트**
        
    - **핵심 이점**: 수개월의 쓸데없는 노력을 아껴주고, 성능을 올리기 위해 **지금 당장 무엇을 해야 하는지(데이터 수집, 파라미터 조정, 모델 변경 등)** 명확한 지침을 제공합니다.

## 2. 모델 평가하기

### 1). 모델 평가가 필요한 이유

- **시각화의 한계:** 입력 변수(Feature)가 1~2개일 때는 그래프를 그려 오버피팅(Overfitting) 여부를 한눈에 볼 수 있지만, **특징이 여러 개(4차원 이상)가 되면 시각화가 불가능**해집니다.
    
- **학습 데이터 오차의 착시:** 4차 다항식 모델처럼 훈련 데이터에 너무 완벽하게 일치하더라도, **훈련 데이터에만 오버피팅되어 새로운 입력에 대한 예측 성능이 매우 떨어질 수 있습니다.**
    
---
### 2). Train / Test 데이터 분할 (Data Splitting)

전체 데이터를 단일 파라미터 학습에 전부 쓰지 않고, 보통 **70:30** 또는 **80:20** 비율로 나눕니다.

```
[ 전체 데이터셋 (Full Dataset) ]
┌───────────────────────────────┬─────────────────────────┐
│     Training Set (70~80%)     │     Test Set (20~30%)   │
│   파라미터(W, B) 학습에 사용  │  모델 성능 최종 검증용  │
└───────────────────────────────┴─────────────────────────┘
```

- **$m_{\text{train}}$**: 훈련 샘플 개수
    
- **$m_{\text{test}}$**: 테스트 샘플 개수
    
---
### 3). 회귀 문제(Regression)에서의 평가 지표

회귀에서는 평균 제곱 오차(MSE, Mean Squared Error)를 기반으로 평가합니다.

#### 훈련 오차 ($J_{\text{train}}$)

$$J_{\text{train}}(W, B) = \frac{1}{2 m_{\text{train}}} \sum_{i=1}^{m_{\text{train}}} \left( f_{W, B}\left(x_{\text{train}}^{(i)}\right) - y_{\text{train}}^{(i)} \right)^2$$

#### 테스트 오차 ($J_{\text{test}}$)

$$J_{\text{test}}(W, B) = \frac{1}{2 m_{\text{test}}} \sum_{i=1}^{m_{\text{test}}} \left( f_{W, B}\left(x_{\text{test}}^{(i)}\right) - y_{\text{test}}^{(i)} \right)^2$$

> ⚠️ **주의 (중요):**
> 
> $J_{\text{train}}$ 및 $J_{\text{test}}$를 계산할 때, 파라미터 학습 시 사용했던 **정규화 항(Regularization term, $\frac{\lambda}{2m}\sum W^2$)은 포함하지 않고 오차만 계산**합니다.

####  오버피팅 진단법

- **$J_{\text{train}}$이 매우 낮고, $J_{\text{test}}$가 매우 높음** $\rightarrow$ 오버피팅 (일반화 실패)
    
---
### 4). 분류 문제(Classification)에서의 평가 지표

분류 문제에서는 2가지 방식으로 오차를 측정할 수 있습니다.

1. **로지스틱 손실 평균 방식:**
    
    각 테스트 샘플의 Logistic Loss를 구해 평균을 냅니다.
    
2. **오분류율 방식 (Misclassification Error, 실무에서 더 흔함):**
    
    예측값 $\hat{y}$ (0 또는 1)와 실제 값 $y$가 일치하지 않는 샘플의 비율(오류율)을 직관적으로 계산합니다.
    

$$J_{\text{test}}(W, B) = \frac{1}{m_{\text{test}}} \sum_{i=1}^{m_{\text{test}}} \mathbb{I}\left(\hat{y}_{\text{test}}^{(i)} \neq y_{\text{test}}^{(i)}\right)$$

_(단, $\mathbb{I}$는 예측이 틀렸을 때 1, 맞았을 때 0을 반환하는 함수)_

## 3. 모델 선택, Training/Cross Validation/Test Sets

### 1). 테스트 세트(Test Set)만으로 모델을 선택하면 안 되는 이유

여러 다항식 모델($d=1, 2, \dots, 10$차) 중 최적의 차수를 찾기 위해 **$J_{\text{test}}$가 가장 낮은 모델을 선택하는 방식은 치명적인 오류**를 범하게 됩니다.

- **원인**: 모델 파라미터 $W, B$를 Train set으로 학습시킨 뒤, 차수 $d$(다항식 차수)라는 **하이퍼파라미터를 Test set을 보고 선택**했기 때문입니다.
    
- **결과**: Test set에 "맞춰진(Fitted)" 모델이 되어, 해당 $J_{\text{test}}$ 값은 실제 새로운 데이터에 대한 성능(일반화 오차)보다 **지나치게 낙관적으로(Optimistically) 평가**됩니다.
    
---
### 2). 올바른 해결책: 데이터를 3개 영역으로 분할 (60 : 20 : 20)

하이퍼파라미터 결정용 데이터와 최종 평가용 데이터를 완전히 분리하기 위해 데이터셋을 **3가지**로 나눕니다.

```
[ 전체 데이터셋 (Full Dataset) ]
┌─────────────────────────────┬───────────────────────┬───────────────────┐
│     Training Set (60%)      │ Validation Set (20%)  │  Test Set (20%)   │
│   파라미터(W, B) 학습용     │ 모델 선택 (d, 구조)   │ 최종 일반화 오차  │
│                             │ (Cross-Validation/Dev)│      평가용       │
└─────────────────────────────┴───────────────────────┴───────────────────┘
```

#### 용어 정리

- **Cross-Validation Set (교차 검증 세트)** = **Validation Set** = **Dev Set (개발 세트)**
    
    - 실무 및 연구진들은 가장 짧은 단어인 **Dev set**이나 **Validation set**이라는 용어를 자주 사용합니다.
        
---
### 3). 3단계 모델 선택 프로세스 (Model Selection Pipeline)

#### 1단계 : Train Set으로 모델 학습

각 후보 모델($d=1, 2, \dots, 10$차 또는 다양한 신경망 아키텍처)의 파라미터 $W, B$를 **Training set**으로 학습시킵니다.

$$\min_{W, B} J(W, B)$$

#### 2단계 : Validation Set으로 최적의 모델(하이퍼파라미터) 선택

**Validation set**의 오차($J_{\text{cv}}$)를 계산하여, 오차가 가장 작은 모델을 최종 아키텍처로 선택합니다.

$$J_{\text{cv}}(W^{(d)}, B^{(d)}) \rightarrow \text{가장 낮은 오차를 가진 모델 선택 (예: } d=4\text{)}$$

#### 3단계 : Test Set으로 공정한 최종 성능 측정

선택된 모델의 최종 일반화 오차(Generalization Error)를 **Test set**으로 단 한 번만 평가합니다.

$$J_{\text{test}}(W^{(4)}, B^{(4)})$$

> 💡 **핵심 원칙**: Test set은 모델의 $W, B$ 학습뿐만 아니라, **아키텍처/하이퍼파라미터 결정 과정에서도 절대 참조되어서는 안 됩니다.**

---

### 4). 신경망(Neural Network) 아키텍처 선택 적용 예시

선형 회귀의 차수($d$)뿐만 아니라, **신경망의 층(Layer) 수나 노드(Unit) 수**를 정할 때도 동일한 원리가 적용됩니다.

```
[모델 1: 소형 NN] ──> J_cv 계산 ┐
[모델 2: 중형 NN] ──> J_cv 계산 ┼──> 가장 J_cv가 낮은 '모델 2' 선택 ──> J_test로 최종 성능 보고
[모델 3: 대형 NN] ──> J_cv 계산 ┘
```

1. **소형/중형/대형 신경망**을 각각 Train set으로 학습
    
2. Validation set 오분류율(Misclassification Rate)이 가장 낮은 신경망 선택
    
3. 해당 신경망의 성능을 **Test set**으로 최종 평가 및 보고

# < Bias and Variance >
- 모델을 선택한 후, 성능을 더 끌어올리기 위한 가장 강력한 진단 도구인 "편향과 분산 (Bias & Variance)"의 개념에 대해 알아봅니다.

## 1. Bias와 Variance 진단하기
### 1).  머신러닝 개발 워크플로우

- **현실적인 머신러닝 개발:** 아이디어를 가지고 모델을 훈련시키면 **첫 번째 시도에 잘 작동하는 경우는 거의 없습니다.**
    
- **핵심 질문:** 모델 성능이 기대에 미치지 못할 때 "성능 개선을 위해 다음에 무엇을 해야 하는가?"를 결정하는 것이 시스템 구축의 핵심입니다.
    
- **해결책:** 편향(Bias)과 분산(Variance)을 진단하면 다음 시도에 대한 명확한 지침을 얻을 수 있습니다.

---
### 2). 시각적 이해 (1차원 특징 $x$ 예시)

변수가 1개일 때는 그래프를 그려서 모델의 상태를 한눈에 파악할 수 있습니다.

|**구분**|**모델 복잡도**|**상태**|**특징**|
|---|---|---|---|
|**High Bias**|$d=1$ (1차 직선)|**Underfitting** (언더피팅)|데이터의 기본 경향도 제대로 따라가지 못함|
|**High Variance**|$d=4$ (4차 다항식)|**Overfitting** (오버피팅)|훈련 데이터점들을 너무 과도하게 맞춰 구불구불함|
|**Just Right**|$d=2$ (2차 다항식)|**Good Fit**|데이터의 흐름을 적절하게 잘 포착함|

> 💡 **한계점:** 입력 변수(Feature)가 많아지면 함수 $f$를 2차원 그래프로 시각화하여 상태를 파악하는 것이 불가능해집니다.

---
### 3). 오차 지표를 통한 체계적 진단

시각화가 불가능한 다차원 문제에서는 훈련 오차($J_{\text{train}}$)와 검증 오차($J_{\text{cv}}$, Cross-Validation Error)를 비교하여 진단합니다.

#### High Bias (편향이 높은 상태 = Underfitting)

- **$J_{\text{train}}$ (훈련 오차):** **높음** (모델이 너무 단순해서 훈련 데이터조차 제대로 맞추지 못함)
    
- **$J_{\text{cv}}$ (검증 오차):** **높음** ($J_{\text{train}}$과 비슷하게 높은 수준)
    
- **진단 핵심:** **$J_{\text{train}}$ 자체가 높은 것**이 편향이 높다는 가장 강력한 신호입니다.
    

#### High Variance (분산이 높은 상태 = Overfitting)

- **$J_{\text{train}}$ (훈련 오차):** **낮음** (훈련 데이터는 완벽하게 암기함)
    
- **$J_{\text{cv}}$ (검증 오차):** **매우 높음** ($J_{\text{cv}} \gg J_{\text{train}}$)
    
- **진단 핵심:** 본 적 없는 데이터($J_{\text{cv}}$)에서의 오차가 이미 본 데이터($J_{\text{train}}$)보다 **훨씬 더 높다**는 점이 분산이 높다는 신호입니다.
    

#### Just Right (적절한 상태)

- **$J_{\text{train}}$ (훈련 오차):** **낮음**
    
- **$J_{\text{cv}}$ (검증 오차):** **낮음** ($J_{\text{train}}$보다 약간 높거나 거의 유사함)
    
---
### 4). 다항식 차수($d$)에 따른 오차 추이 그래프

정규화(Regularization)를 사용하지 않는다고 가정할 때, 다항식 차수 $d$(Horizontal Axis)에 따른 $J_{\text{train}}$과 $J_{\text{cv}}$의 변화 흐름입니다.

```
       오차(Error)
         ▲
         │   \                   /
         │    \      J_cv       /   <-- High Variance (Overfitting) Zone
         │     \  (Validation) /        J_cv >> J_train
         │      \   ________   /
 High Bias Zone  \ /        \ /
 (Underfitting)   ▼          ▼
  J_train High    │  Just    │
                  │  Right   │
                  └──────────┴────────────────► 다항식 차수 d (Degree of Polynomial)
                  │          │
                  └──────────┴────────────────► J_train (Training Error)
```

1. **$J_{\text{train}}$ 곡선:** 차수 $d$가 증가할수록 모델 표현력이 높아져 훈련 데이터에 더 잘 맞추므로 우하향(지속적 감소)합니다.
    
2. **$J_{\text{cv}}$ 곡선:**
    
    - $d$가 매우 작을 때(Left): 언더피팅으로 인해 $J_{\text{cv}}$가 높음.
        
    - $d$가 중간일 때(Center): 가장 적절한 일반화 성능을 보이며 **$J_{\text{cv}}$ 최저점** 형성.
        
    - $d$가 매우 크기 시작할 때(Right): 오버피팅으로 인해 $J_{\text{cv}}$가 다시 **우상향(상승)**.
        
---
### 5). 수학적 요약 및 특수 케이스

#### 진단 공식 요약

- **High Bias (Underfit):** $J_{\text{train}}$이 높음 (대체로 $J_{\text{cv}} \approx J_{\text{train}}$)
    
- **High Variance (Overfit):** $J_{\text{cv}} \gg J_{\text{train}}$ (기호 `>>`는 "much greater than", 즉 **훨씬 크다**를 의미)
    

#### 특수 케이스: High Bias와 High Variance가 동시에 발생하는 경우

- **발생 조건:** 선형 회귀(Linear Regression)에서는 흔치 않지만, 복잡한 **신경망(Neural Network)** 구조에서 가끔 발생합니다.
    
- **직관적 설명:** 입력 공간의 일부 영역에서는 모델이 너무 복잡하여 **오버피팅**되고, 동시에 다른 영역에서는 제대로 패턴을 잡지 못해 **언더피팅**되는 경우입니다.
    
- **진단 지표:** $J_{\text{train}}$도 높고, $J_{\text{cv}}$는 그보다 훨씬 더 높은 경우 ($J_{\text{cv}} \gg J_{\text{train}}$ 이면서 $J_{\text{train}}$도 나쁨).
    
---
### 6). 결론 

- 머신러닝 시스템 개선 시 최우선 과제는 현재 모델이 **High Bias(언더피팅)** 문제인지, **High Variance(오버피팅)** 문제인지를 판별하는 것입니다.

## 2. 정규화와 Bias/Variance

### 1). 정규화 파라미터 $\lambda$의 역할

- **정규화(Regularization) 목적:** 모델의 가중치 $w$가 너무 커지지 않도록 억제하여 오버피팅을 방지함.
    
- **학습 목표:** $\lambda$ 값의 선택이 모델의 편향(Bias)과 분산(Variance)에 어떠한 영향을 미치는지 이해하고, 교차 검증(Cross-Validation)을 통해 최적의 $\lambda$를 찾는 법을 배움.
    
---
### 2). $\lambda$ 값에 따른 3가지 극단적/이상적 상황

4차 다항식($d=4$) 모델을 예시로 들었을 때 $\lambda$에 따른 모델의 변화입니다.

| **구분**            | **λ 값**                   | **모델 상태**               | **오차 특징**                                                                      | **직관적 결과**                                              |
| ----------------- | ------------------------- | ----------------------- | ------------------------------------------------------------------------------ | ------------------------------------------------------- |
| **High Bias**     | 매우 큼 ($\lambda = 10,000$) | **Underfitting** (언더피팅) | $J_{\text{train}}$ 높음<br><br>  <br><br>$J_{\text{cv}}$ 높음                      | $w$ 파라미터들이 0에 가까워져 $f(x) \approx b$ 형태의 평평한 수직/수평 직선이 됨 |
| **High Variance** | 매우 작음 ($\lambda = 0$)     | **Overfitting** (오버피팅)  | $J_{\text{train}}$ 매우 낮음<br><br>  <br><br>$J_{\text{cv}} \gg J_{\text{train}}$ | 정규화 효과가 전혀 없어 구불구불하게 훈련 데이터를 외워버림                       |
| **Just Right**    | 적절한 중간값                   | **Good Fit**            | $J_{\text{train}}$ 낮음<br><br>  <br><br>$J_{\text{cv}}$ 낮음                      | 패턴을 적절히 학습하여 새로운 데이터에도 높은 예측력 보임                        |
|                   |                           |                         |                                                                                |                                                         |

---

### 3). 교차 검증(CV)을 통한 최적의 $\lambda$ 선택 절차

**1.후보 lambda 설정 및 모델 훈련:**Step 1.

$\lambda = 0, 0.01, 0.02, 0.04, 0.08, \dots, 10$ 과 같이 대략 2배씩 늘려가며 다양한 $\lambda$ 후보군을 설정합니다. 각 $\lambda$로 비용함수를 최적화하여 파라미터 세트 $(w^{(1)}, b^{(1)}), (w^{(2)}, b^{(2)}), \dots$ 를 학습시킵니다.

**2.교차 검증 오차(J_cv) 계산:**Step 2.

학습된 각 파라미터 세트에 대해 **검증 데이터셋(CV Set)**에서의 오차인 $J_{\text{cv}}(w^{(i)}, b^{(i)})$를 각각 계산합니다.

**3.최적의 lambda 선택:**Step 3.

계산된 $J_{\text{cv}}$ 값들 중 **가장 작은 오차를 보인 모델**(예: 5번째 모델의 $J_{\text{cv}}(w^{(5)}, b^{(5)})$)의 $\lambda$를 최종 정규화 파라미터로 선택합니다.

**4.테스트 데이터셋 평가:**Step 4.

선택된 최종 모델의 일반화 성능을 보고하기 위해 **테스트 데이터셋(Test Set)**에서 $J_{\text{test}}(w^{(5)}, b^{(5)})$를 평가합니다.

---

### 4). $\lambda$ 변화에 따른 오차 추이 그래프

$\lambda$가 커짐에 따라 훈련 오차($J_{\text{train}}$)와 검증 오차($J_{\text{cv}}$)는 아래와 같은 형태를 보입니다.

```
       오차(Error)
         ▲
         │   \                   /
         │    \      J_cv       /   <-- High Bias (Underfitting) Zone
         │     \  (Validation) /        J_train & J_cv 모두 높음
         │      \   ________   /
 High Variance   \ /        \ /
 (Overfitting)    ▼          ▼
 J_cv >> J_train  │  Just    │
                  │  Right   │
                  └──────────┴────────────────► 정규화 파라미터 λ (Lambda)
                  │          │
                  └──────────┴────────────────► J_train (Training Error) [우상향]
```

1. **$J_{\text{train}}$ (훈련 오차):** $\lambda$가 증가할수록 가중치 $w$를 줄이는 데 집중하므로, 훈련 데이터를 맞추는 능력은 떨어져 우상향(지속적 증가)합니다.
    
2. **$J_{\text{cv}}$ (검증 오차):**
    
    - $\lambda$가 너무 작을 때(Left): 오버피팅(High Variance)으로 인해 $J_{\text{cv}}$가 높음.
        
    - $\lambda$가 적절할 때(Center): **$J_{\text{cv}}$가 가장 낮은 최적 지점** 형성.
        
    - $\lambda$가 너무 클 때(Right): 언더피팅(High Bias)으로 인해 $J_{\text{cv}}$가 다시 높아짐.
        
---

### 5). 거울상(Mirror Image) 비교

이전 강의에서 다룬 **다항식 차수($d$) 그래프**와 이번 **정규화 파라미터($\lambda$) 그래프**는 서로 거울에 비친 반대 형태(Mirror Image)를 띱니다.

|**X축 변수**|**왼쪽 (Small Value)**|**오른쪽 (Large Value)**|
|---|---|---|
|**다항식 차수 ($d$)**|High Bias (언더피팅)|High Variance (오버피팅)|
|**정규화 파라미터 ($\lambda$)**|High Variance (오버피팅)|High Bias (언더피팅)|

---

### 6). 결론

- $J_{\text{train}}$이 높으면 High Bias, $J_{\text{cv}}$가 $J_{\text{train}}$보다 훨씬 높으면 High Variance로 진단할 수 있습니다.

## 3. 성능의 Baseline Level 설정하기

### 1). 개요: 베이스라인 성능(Baseline)이 필요한 이유

- **기존 문제점:** $J_{\text{train}}$(훈련 오차)이 10%가 나왔을 때, 이 값만 보고 바로 "훈련 오차가 높으니 High Bias(언더피팅)이다"라고 결론을 내리기 어렵습니다.
    
- **이유:** 배경 소음이 심한 음성 인식처럼 **데이터 자체에 노이즈가 많아 인간조차 완벽히 맞추지 못하는 문제**가 존재하기 때문입니다.
    
- **해결책:** 모델이 현실적으로 달성할 수 있는 목표 오차 수준인 베이스라인 성능(Baseline Level of Performance)을 설정하고, 이와 비교하여 진단해야 합니다.

---
### 2). 베이스라인 성능(Baseline)을 설정하는 3가지 방법

1. **인간 수준의 성능 (Human-level Performance):**
    
    - 음성, 이미지, 텍스트와 같은 **비구조화 데이터(Unstructured Data)** 처리 문제에서 가장 흔히 사용됩니다.
        
2. **기존/경쟁 알고리즘 성능 (Competing/Previous Algorithm):**
    
    - 이미 구축된 이전 버전 모델이나 타사 알고리즘의 오차율을 기준으로 삼습니다.
        
3. **이전 경험에 기반한 추정 (Prior Experience):**
    
    - 유사 문제 해결 경험에 기초하여 달성 가능한 현실적인 오차율을 예상합니다.
        

> 💡 만약 완벽한 예측이 가능한 정제된 데이터라면 베이스라인 오차율은 0%가 될 수도 있습니다.
> 

---
### 3). 베이스라인 기준 편향 & 분산 진단법

진단 시에는 다음 두 가지 격차(Gap)를 측정하는 것이 핵심입니다.

- **Gap 1 (Bias Indicator):** $\text{Baseline Performance}$ 와 $J_{\text{train}}$ 간의 격차
    
- **Gap 2 (Variance Indicator):** $J_{\text{train}}$ 과 $J_{\text{cv}}$ 간의 격차

## 4. 학습 커브

### 1).  학습 곡선(Learning Curve)이란?

- **정의:** 훈련 데이터의 크기($m_{\text{train}}$)에 따른 훈련 오차($J_{\text{train}}$)와 교차 검증 오차($J_{\text{cv}}$)의 변화를 나타낸 그래프입니다.
    
- **주요 목적:** 더 많은 데이터를 수집하는 것이 실제로 모델 성능을 향상시킬 수 있는지 판단하는 직관적 도구입니다.
    
---
### 2). 일반적인 학습 곡선의 동작 원리

```
 오차(Error)
   ▲
   │   \ 
   │    \   J_cv (Validation Error)  [데이터가 많을수록 모델이 일반화되며 감소]
   │     \______________________
   │      ______________________  <-- Typical Gap (J_cv >= J_train)
   │     / 
   │    /   J_train (Training Error) [데이터가 많을수록 완벽히 맞추기 어려워 증가]
   └────────────────────────────────► 훈련 데이터 크기 (m_train)
```

1. **$J_{\text{train}}$ (훈련 오차):**
    
    - 데이터가 1~3개로 적을 때는 직선/곡선으로 100% 맞추기 쉬워 오차가 $0$에 가깝습니다.
        
    - 데이터 크기($m$)가 커질수록 모든 포인트를 완벽히 맞추기가 어려워지므로 **훈련 오차는 상승**합니다.
        
2. **$J_{\text{cv}}$ (검증 오차):**
    
    - 데이터가 늘어날수록 모델이 더 일반화된 패턴을 학습하므로 **검증 오차는 감소**합니다.
        
3. **일반적 특징:** 파라미터를 훈련 세트에 맞추기 때문에 항상 **$J_{\text{cv}} \ge J_{\text{train}}$** 형태를 띱니다.
    
---
### 3). High Bias vs High Variance 학습 곡선 비교

#### (1). High Bias (언더피팅, 모델이 너무 단순한 경우)

```
 오차(Error)
   ▲
   │   \________________________ J_cv (High)
   │    ________________________ J_train (High)
   │   - - - - - - - - - - - - - Baseline (Human-level)
   └────────────────────────────────► m_train
```

- **특징:**
    
    - $J_{\text{train}}$과 $J_{\text{cv}}$ 모두 높고, 거의 수평으로 정체(Plateau)합니다.
        
    - 데이터가 늘어나도 단순한 모델(예: 1차 직선)의 한계 때문에 성능이 개선되지 않습니다.
        
    - $J_{\text{train}}$과 Baseline(인간 수준) 사이의 격차가 큽니다.
        
- **핵심 결론:** **High Bias 상태에서는 훈련 데이터를 더 많이 수집해도 오차가 줄어들지 않습니다 (도움이 안 됨).**
    

#### (2) High Variance (오버피팅, 모델이 너무 복잡한 경우)

```
 오차(Error)
   ▲
   │   \
   │    \____________________ J_cv (지속 감소)
   │               Large Gap
   │    _____________________ J_train (천천히 상승)
   │   /
   └────────────────────────────────► m_train
```

- **특징:**
    
    - $J_{\text{train}}$은 매우 낮고, $J_{\text{cv}}$는 상대적으로 매우 높아 두 곡선 사이에 큰 격차(Large Gap)가 존재합니다.
        
    - 데이터 크기($m$)를 계속 늘려 x축 오른쪽으로 확장(Extrapolate)하면, $J_{\text{cv}}$가 지속적으로 하강하며 $J_{\text{train}}$ 및 Baseline에 가까워집니다.
        
- **핵심 결론:** **High Variance 상태에서는 훈련 데이터를 더 많이 수집하는 것이 성능 개선에 매우 효과적입니다.**
    
---
### 4). 실전 활용법 및 단점

- **실전 그리는 방법:** 훈련 데이터가 1,000개라면, 100개, 200개, 300개 등 서브셋(Subset) 크기를 점진적으로 늘려가며 여러 번 모델을 학습시키고 $J_{\text{train}}$과 $J_{\text{cv}}$를 좌표에 찍어 연결합니다.
    
- **실제 적용 시 단점:** 크기별로 수많은 모델을 재학습시켜야 하므로 **계산 비용(Computational Cost)이 큼**:
    
    - 따라서 매번 실제 그래프를 그리기보다는 "이 머릿속의 시각적 프레임워크(Mental Picture)"를 활용하여 데이터 추가 수집 여부를 결정하는 편입니다.
        
---
### 5). 핵심 요약 Table

|**상태**|**Jtrain​ & Jcv​ 특징**|**데이터 추가 수집의 효과**|
|---|---|---|
|**High Bias**|두 오차 모두 높으며 빠르게 정체됨 (Plateau)|**효과 없음** (모델 복잡도 증가 필요)|
|**High Variance**|두 오차 간 격차(Gap)가 매우 큼|**매우 효과적** (일반화 성능 향상)|

### 6). 결론 
- 데이터를 수집하기 위해 많은 시간과 비용을 투자하기 전에, 현재 모델이 **High Bias**인지 **High Variance**인지 학습 곡선 개념으로 진단하는 것이 우선입니다.

## 5. Deciding what to try next revisited
- 머신러닝 모델의 성능이 만족스럽지 않을 때 무작위로 해결책을 시도하는 대신, 편향(Bias)과 **분산(Variance)** 진단 결과에 따라 **어떤 조치를 취해야 하는지 매핑하는 실전 지침**

### 1). 개요: 진단 기반의 의사결정

머신러닝 모델(예: 주택 가격 예측 선형 회귀)의 예측 오차가 수용할 수 없을 정도로 클 때, 우리가 고려할 수 있는 6가지 해결법이 있습니다. 이 6가지 해결책은 **High Variance(오버피팅)를 해결하는 3가지**와 **High Bias(언더피팅)를 해결하는 3가지**로 명확히 나뉩니다.

### 2). 문제 원인별 6가지 해결책 분류

|**문제 상태**|**시도할 해결책 (Action Plan)**|**작동 원리 및 이유**|
|---|---|---|
|**High Variance**<br><br>  <br><br>(오버피팅 / 모델이 너무 복잡함)|**1. 더 많은 훈련 데이터 수집**<br><br>  <br><br>(Get more training examples)|복잡한 모델을 더 안정적으로 학습시키고 일반화 능력을 향상시킴|
||**2. 특성(Feature) 개수 줄이기**<br><br>  <br><br>(Try a smaller set of features)|불필요하거나 중복된 특성을 제거하여 모델의 과도한 유연성을 제한함|
||**3. 정규화 파라미터 $\lambda$ 증가**<br><br>  <br><br>(Increase $\lambda$)|가중치 $w$에 대한 제약을 강화하여 곡선을 더 매끄럽게(Smooth) 만듦|
|**High Bias**<br><br>  <br><br>(언더피팅 / 모델이 너무 단순함)|**4. 추가 특성(Feature) 도입**<br><br>  <br><br>(Get additional features)|주택 크기 외에 침실 수, 층수, 연식 등 필요한 유용한 정보를 모델에 추가함|
||**5. 다항식 특성(Polynomial) 추가**<br><br>  <br><br>(Add polynomial features)|$x^2, x^3$ 등을 추가하여 모델의 표현력과 유연성(Flexibility)을 높임|
||**6. 정규화 파라미터 $\lambda$ 감소**<br><br>  <br><br>(Decrease $\lambda$)|정규화 제약을 줄여 훈련 데이터의 패턴을 더 잘 맞추도록 함|

### 3). 핵심 규칙 요약 (Mental Model)

```
                            ┌──► 훈련 데이터 추가 (More Data)
    ┌───────────────────┐   │
    │   High Variance   ├───┼──► 특성 개수 축소 (Fewer Features)
    │ (Fix Overfitting) │   │
    └───────────────────┘   └──► 정규화 강화 (Increase λ)
    
    
                            ┌──► 추가 특성 도입 (More Features)
    ┌───────────────────┐   │
    │     High Bias     ├───┼──► 다항식 특성 추가 (Polynomials)
    │(Fix Underfitting) │   │
    └───────────────────┘   └──► 정규화 완화 (Decrease λ)
```

> ⚠️ **주의사항 (Mistake to Avoid):**
> 
> High Bias(언더피팅) 문제를 해결하기 위해 **훈련 데이터 크기를 줄이는 시도는 절대 하면 안 됩니다.** 훈련 데이터셋을 의도적으로 줄이면 훈련 오차는 줄어들 수 있으나, 교차 검증 오차($J_{\text{cv}}$)와 전체적인 일반화 성능은 오히려 더 악화됩니다.

### 4). 실무 조언

- **"배우는 데는 짧은 시간이 걸리지만, 정통하기까지는 평생이 걸리는 개념"**
    
    - 편향과 분산은 머신러닝의 가장 강력하고 기초적인 핵심 개념입니다.
        
    - 단순히 이론을 아는 것을 넘어, 수많은 프로젝트 실무와 반복 연습을 통해 모델의 상태를 감지하고 체계적으로 대처하는 능력을 기르는 것이 중요합니다.

## 6. Bias / Variance and Neural Networks
- 딥러닝과 신경망(Neural Networks)의 발전이 전통적인 머신러닝의 **"편향-분산 트레이드오프(Bias-Variance Trade-off)"** 패러다임을 어떻게 바꾸었는지, 그리고 신경망을 활용한 현대적인 체계적 모델 개선 프로세스(Recipe)
### 1). 기존 머신러닝 vs 딥러닝 시대의 패러다임 변화

#### (1) 전통적 머신러닝: Bias-Variance Trade-off

- 과거에는 모델이 너무 단순하면 High Bias(언더피팅)가 발생하고, 너무 복잡하면 High Variance(오버피팅)가 발생했습니다.
    
- 엔지니어들은 모델의 복잡도(다항식 차수, 정규화 파라미터 $\lambda$)를 세심하게 조절하여 편향과 분산 사이의 아슬아슬한 균형점(Trade-off)을 찾아야 했습니다.
    

#### (2) 신경망(Neural Network) 시대: 트레이드오프의 극복

- 충분히 큰 신경망(Large Neural Network)은 "Low-Bias Machine(낮은 편향을 만드는 도구)"입니다.
    
- 데이터셋이 터무니없이 거대하지 않다면, 신경망의 크기를 키우는 것만으로 훈련 오차를 거의 항상 $0$에 가깝게 낮출 수 있습니다.
    
- 따라서 편향과 분산 중 하나를 희생하지 않고도 **두 문제를 독립적으로 하나씩 해결**할 수 있는 새로운 접근법(Recipe)이 가능해졌습니다.
    
---
### 2). 신경망 학습의 현대적 해결 루프 (The Basic Recipe)

신경망 모델을 개선할 때는 아래 흐름도에 따라 순차적이고 체계적으로 문제를 해결합니다.

```
                  [1. 모델 훈련]
                         │
                         ▼
             훈련 세트 오차(J_train)가 높은가?
             (High Bias 문제인가?)
               /               \
             YES                NO
             /                   \
   [조치] 신경망 크기 확대           [2단계 이동]
   (더 많은 레이어 / 유닛)                │
             │                           ▼
             └──────► 재훈련 ◄──────── 검증 세트 오차(J_cv)가 높은가?
                                     (High Variance 문제인가?)
                                       /               \
                                     YES                NO
                                     /                   \
                           [조치] 더 많은 데이터 수집      [완성!]
                           (또는 적절한 정규화)       성능 목표 달성
```

#### 1단계 훈련 세트 오차($J_{\text{train}}$) 체크 $\rightarrow$ Bias 해결

- **질문:** 인간 수준(Baseline) 대비 훈련 오차가 높은가?
    
- **해결책:** 더 큰 신경망(Bigger Neural Network)을 사용합니다.
    
    - 은닉층(Hidden Layer) 개수를 늘리거나, 레이어당 은닉 유닛(Hidden Unit) 수를 늘립니다.
        
    - 훈련 오차가 목표 수준에 도달할 때까지 신경망 크기를 지속적으로 키웁니다.
        

#### 2단계 검증 세트 오차($J_{\text{cv}}$) 체크 $\rightarrow$ Variance 해결

- **질문:** $J_{\text{train}}$은 낮으나 $J_{\text{cv}}$와의 격차가 큰가?
    
- **해결책:** 더 많은 데이터(More Data)를 수집하거나 정규화(Regularization)를 적용합니다.
    
- **반복:** 검증 오차가 만족스러워질 때까지 위 루프를 반복합니다.
    
---
### 3). 신경망 크기와 오버피팅에 대한 오해와 진실

- **질문:** "신경망이 너무 크면 High Variance(오버피팅) 문제가 생기지 않나요?"
    
- **진실:** **적절한 정규화(Regularization)가 적용되어 있다면, 더 큰 신경망은 작은 신경망보다 거의 항상 같거나 더 나은 성능을 냅니다.**
    
- **단점(Caveat):** 크기가 커질수록 오버피팅이 발생하는 것이 아니라, **계산 비용(Computational Cost)이 커져 훈련 및 추론 속도가 느려집니다.**
    
---
### 4). TensorFlow에서의 L2 정규화 구현법

신경망의 가중치 $w$에 제약을 주어 오버피팅을 방지하는 L2 정규화 비용 함수:

$$J(w,b) = \text{Average Loss} + \frac{\lambda}{2m} \sum w^2$$

#### TensorFlow 코드 예시

비정규화 모델 설정 코드에 `kernel_regularizer` 옵션을 추가합니다.



```Python
import tensorflow as tf
from tensorflow.keras import Sequential
from tensorflow.keras.layers import Dense
from tensorflow.keras.regularizers import l2

# L2 정규화가 적용된 신경망 모델
model = Sequential([
    Dense(units=25, activation='relu', kernel_regularizer=l2(0.01)),  # lambda = 0.01
    Dense(units=15, activation='relu', kernel_regularizer=l2(0.01)),
    Dense(units=1,  activation='sigmoid')
])
```

_(참고: 편향 $b$에 대해서는 보통 정규화를 적용하지 않으며, 적용하더라도 결과에 큰 차이가 없습니다.)_

# < ML development Process > 

## 1. 머신러닝 개발의 반복적 루프
- 머신러닝 시스템을 구축할 때 겪게 되는 반복적 개발 프로세스(Iterative Loop)의 구조와, **이메일 스팸 분류기(Spam Classifier)** 예시를 바탕으로 아이디어를 효율적으로 선택하는 원리

### 1). 머신러닝 개발의 핵심: 반복적 루프 (Iterative Loop)

머신러닝 모델 개발은 단 한 번에 완성되지 않으며, 아래의 3단계 루프를 계속해서 돌며 성능을 개선해 나가는 과정입니다.

```
                  ┌───────────────────────────────┐
                  │ 1. 시스템 아키텍처/모델 결정   │
                  │    (Data, Hyperparameters)    │
                  └──────────────┬────────────────┘
                                 │
                                 ▼
                  ┌───────────────────────────────┐
                  │ 2. 모델 구현 및 훈련 (Train)  │
                  └──────────────┬────────────────┘
                                 │
                                 ▼
                  ┌───────────────────────────────┐
                  │ 3. 진단 도구 실행 (Diagnostics)│
                  │    (Bias/Variance, Error Anal)│
                  └──────────────┬────────────────┘
                                 │
                                 └───► [개선 방향 수립 후 1단계로 반복]
```

1. **시작 (Choose Architecture):** 데이터, 알고리즘, 하이퍼파라미터 등 전체적인 구조를 결정합니다.
    
2. **훈련 (Train Model):** 처음 만든 모델은 대부분 한 번에 원하는 성능을 내지 못합니다.
    
3. **진단 (Diagnostics):** 편향/분산(Bias/Variance) 분석 및 오류 분석(Error Analysis)을 실행합니다.
    
4. **개선 및 반복 (Iterate):** 진단 결과에서 얻은 인사이트를 바탕으로 모델 크기, 정규화($\lambda$), 데이터/특성(Feature) 조절 등의 의사결정을 내리고 다음 루프를 실행합니다.
    
---
### 2). 실전 예시: 이메일 스팸 분류기 (Spam Classifier)

스팸 이메일과 일반 이메일을 구별하는 텍스트 분류(Text Classification) 문제를 다룹니다.

#### (1) 특성 벡터(Feature Vector) 구성 방법

- 사전(Dictionary)에서 자주 사용되는 단어 10,000개를 선정합니다.
    
- 입력 이메일에 해당 단어가 포함되어 있는지 여부(0 또는 1)로 10,000차원의 특성 벡터 $x$를 생성합니다. (단어의 출현 빈도수를 카운트하는 방식도 가능)
    
- 이 특성 벡터 $x$를 입력으로 하여 로지스틱 회귀(Logistic Regression)나 신경망(Neural Network)을 학습시켜 스팸 여부 $y \in {0, 1}$를 예측합니다.
    
---
### 3). 성능 개선을 위한 다양한 아이디어와 우선순위 결정

초기 모델의 성능이 불만족스러울 때 떠올릴 수 있는 개선 아이디어들:

- **데이터 추가 수집:** 가짜 이메일 주소를 노출시켜 스팸을 수집하는 **허니팟(Honeypot)** 프로젝트 진행
    
- **이메일 헤더(Header) 기반 특성:** 이메일이 거쳐온 서버 경로(Routing) 정보를 활용한 특성 설계
    
- **텍스트 처리 고도화:** 단어의 어근 추출(예: `discount`와 `discounting`을 동일 단어로 취급)
    
- **의도적 오탈자 감지:** 스팸 필터를 피하기 위해 스패머들이 쓰는 변형 단어(`w4tches`, `m0rtgage` 등) 감지 알고리즘 구현
    

> 💡 **진단(Diagnostics)이 중요한 이유**
> 
> 무작정 수개월 동안 허니팟 데이터를 수집하더라도, 만약 현재 모델이 **High Bias(언더피팅)** 문제라면 성능 개선 효과는 거의 없습니다.
> 
> 반대로 **High Variance(오버피팅)** 문제라면 데이터 수집이 매우 큰 효과를 냅니다. 진단 도구를 통해 **올바른 방향을 선택하면 개발 속도를 10배 이상 단축**할 수 있습니다.

## 2. Error 분석
- 머신러닝 개발 생산성을 극대화하는 핵심 도구인 **오류 분석(Error Analysis)의 개념, 수립 방법, 실전 적용 사례 및 한계점**
### 1). 오류 분석(Error Analysis)이란?

오류 분석은 **교차 검증 세트(Cross-Validation Set)에서 모델이 잘못 예측한 사례들을 사람이 직접 수동으로 살펴보고**, 오류의 공통적인 원인과 패턴을 파악하여 **다음에 어떤 시도를 해야 하는지 방향성을 잡는 기법**입니다.

- **우선순위:** 진단 도구 중 **1위가 편향/분산 분석**이라면, **2위는 오류 분석**이라고 할 수 있을 만큼 실무에서 강력한 성능을 발휘합니다.
    
---
### 2). 오류 분석 수행 절차 (실전 예시: 스팸 분류기)

만약 전체 $500$개의 교차 검증 데이터 중 모델이 **$100$개의 이메일을 오분류**했다고 가정해 봅시다.

#### 1단계: 오류 유형 카테고리화 및 분류 수작업 (Counting Errors)

오류가 난 100개의 사례를 직접 열어보고 공통 주제/속성별로 분류하여 개수를 셉니다.

|**오류 카테고리**|**오분류 개수**|**의미 분석 및 우선순위 판단**|
|---|---|---|
|**의약품 판매 스팸 (Pharma)**|**21개**|**최우선 해결 과제** (큰 오차 비중)|
|**비밀번호 탈취/피싱 (Phishing)**|**18개**|**최우선 해결 과제** (큰 오차 비중)|
|**이상한 경로/헤더 (Unusual Routing)**|7개|중간 비중|
|**의도적 오탈자 (Misspellings)**|**3개**|**우선순위 낮음** (해결해도 전체 성능 개선 미비)|
|**이미지 매개 스팸 (Embedded Image)**|일부|-|

> 💡 **주의사항 (Key Notes):**
> 
> 1. **상호 배타적이지 않음 (Overlapping):** 하나의 이메일이 "의약품 스팸"이면서 동시에 "의도적 오탈자"를 포함할 수 있으므로, 카테고리는 중복 집계될 수 있습니다.
>     
> 2. **표본 추출 (Sampling):** 오분류된 데이터가 1,000개 이상으로 너무 많다면, 전부 보는 대신 **100~200개 정도를 무작위 표본 추출(Random Sampling)**하여 분석해도 충분한 통계적 직관을 얻을 수 있습니다.
>     

---
### 3). 오류 분석이 주는 실무적 이점

#### (1) 헛수고 방지 (Time-Saving)

- 앤드류 응 교수님 본인도 과거 스팸 이메일의 의도적 오탈자(`w4tches`, `m0rtgage` 등)를 잡아내는 정교한 알고리즘을 구축하는 데 수개월을 보냈으나, 나중에 분석해 보니 전체 오류의 3%에 불과해 영향력이 극히 적었다고 고백했습니다.
    
- 오류 분석을 통해 **효과가 미미한 작업에 몇 달을 낭비하는 위험을 방지**할 수 있습니다.
    

#### (2) 구체적인 해결책 제시 (Targeted Action)

- **타겟팅 데이터 수집:** 무작정 전체 데이터를 모으는 대신, 오차가 큰 **"의약품 스팸"이나 "피싱 이메일" 데이터만 집중 수집**합니다.
    
- **맞춤형 특성(Feature) 엔지니어링:** 특정 약물 명칭 키워드 특성을 추가하거나, 피싱 의심 URL 링크를 감지하는 특화 로직을 개발합니다.
    
---
### 4). 오류 분석의 한계점

- **인간의 직관이 작동하는 문제에 유용:** 사람이 직접 보고 정답과 오답을 판별할 수 있는 영역(이미지 분류, 음성 인식, 텍스트 스팸 분류 등)에서는 매우 강력합니다.
    
- **인간이 판단하기 어려운 문제에는 한계 존재:** 웹사이트 광고 클릭률 예측(CTR), 복잡한 추천 시스템 등 사람조차 정답 예측 구조를 쉽게 알 수 없는 분야에서는 오류 분석을 적용하기 어렵습니다.

## 3. 데이터 추가
- 효율적인 데이터 수집, 데이터 증강(Augmentation), 데이터 합성(Synthesis), 그리고 데이터 중심(Data-centric) 접근법

### 1). 데이터 추가의 효율적 전략: 타겟팅 수집 (Targeted Data Collection)

무작정 모든 종류의 데이터를 수집하는 것은 시간과 비용이 많이 듭니다.

- **핵심 전략:** 오류 분석(Error Analysis) 결과를 바탕으로 **모델이 자주 틀리는 특정 유형의 데이터만 집중적으로 추가 수집**합니다.
    
- **예시:** 스팸 분류기에서 "의약품 스팸(Pharma Spam)"의 오차율이 높다면, 레이블이 지정되지 않은 데이터(Unlabeled Data)에서 의약품 관련 이메일만 골라내어 레이블을 붙입니다.
    
- **효과:** 최소한의 비용과 노력으로 모델의 성능을 비약적으로 상승시킬 수 있습니다.
    
---
### 2). 데이터 증강 (Data Augmentation)

기존 학습 데이터를 변형(Distortion/Transformation)하여 동일한 라벨을 가진 새로운 데이터를 만들어내는 기법입니다. 주로 **이미지 및 오디오 데이터**에 매우 효과적입니다.

#### (1) 이미지 데이터 증강 (OCR 예시)

- 글자 'A' 이미지를 **회전(Rotate), 확대/축소(Enlarge/Shrink), 명암 조절(Contrast), 그리드 변형(Warping)** 등을 적용해 고품질의 추가 학습 사례를 생성합니다.
    

#### (2) 오디오 데이터 증강 (음성 인식 예시)

- 기존의 깨끗한 음성 데이터("What is today's weather?")에 다양한 배경 소음(**군중 소음, 차 안 소음, 열악한 통화 품질 소음**)을 합성하여 실제 환경에 강인한 모델을 만듭니다.
    

#### (3) 데이터 증강 적용 시 주의사항

- **테스트 세트를 반영하는 노이즈 사용:** 테스트 환경에서 실제로 발생할 법한 노이즈나 변형을 적용해야 효과가 있습니다.
    
- **무의미한 픽셀 단위 픽셀 노이즈(Per-pixel random noise) 지양:** 실제 환경에서 접하기 힘든 완전 무작위 노이즈는 모델 성능 개선에 도움이 되지 않습니다.

---
### 3). 데이터 합성 (Data Synthesis)

기존 데이터를 변형하는 것이 아니라, **아예 백지 상태에서 인공적인 새로운 데이터를 만들어내는 기법**입니다.

- **사진 속 글자 인식(Photo OCR) 예시:**
    
    - 컴퓨터의 다양한 폰트(Font)와 색상, 배경을 조합하여 수많은 텍스트 이미지를 자동 생성(Screenshot)합니다.
        
    - 코드를 작성하는 데 초기 노력이 들지만, 컴퓨터 비전(Computer Vision) 분야에서 엄청난 양의 학습 데이터를 손쉽게 확보하는 강력한 방법입니다.
        

### 4). 모델 중심(Model-centric) vs 데이터 중심(Data-centric) AI

|**구분**|**모델 중심 접근법 (Model-centric)**|**데이터 중심 접근법 (Data-centric)**|
|---|---|---|
|**주요 작업**|데이터를 고정한 채 **알고리즘 코드/모델 구조/하이퍼파라미터** 개선|코드/알고리즘을 고정한 채 **데이터의 품질과 양**을 개선|
|**특징**|지난 수십 년간 학계 및 연구 분야의 주류 방식|이미 뛰어난 알고리즘이 많아진 **현대 머신러닝 실무의 핵심**|
|**적용 기법**|신경망 구조 변경, 정규화($\lambda$) 조절 등|타겟팅 수집, 데이터 증강, 데이터 합성 등|

## 4. Transfer Learning
- Using Data From a different task
- 데이터가 부족한 상황에서 아주 유용하게 쓰이는 **전이 학습(Transfer Learning)의 개념, 동작 방식, 실전 선택지, 그리고 작동 원리**


### 1). 전이 학습(Transfer Learning)이란?

전이 학습은 **대규모 데이터셋으로 미리 학습된 모델(Pre-trained Model)의 지식(가중치/파라미터)을 가져와, 데이터가 적은 나의 애플리케이션에 재활용하여 성능을 크게 끌어올리는 기법**입니다.

- **사용 목적:** 내가 가진 라벨링 데이터가 몇 십~몇 천 개 수준으로 매우 적을 때 유용합니다.
    
- **주요 용도:** 이미지 인식, 음성 인식, 자연어 처리(NLP) 등 전 분야에서 널리 활용됩니다.
    
---
### 2). 전이 학습의 2단계 프로세스

```
[1단계: 사전 학습 (Pre-training)]
 대규모 데이터 (예: 100만 개 이미지, 1,000개 클래스)
 ──► 신경망 전체 학습 ──► 파라미터 (W1, b1 ~ W5, b5) 추출

                           │
                           ▼
[2단계: 미세 조정 (Fine-tuning)]
 내 데이터셋 (예: 손글씨 0~9 인식, 10개 클래스)
 ──► 마지막 출력층 변경 (1,000개 ──► 10개)
 ──► 가중치 재활용 및 재학습 진행
```

1. **사전 학습 (Supervised Pre-training):**
    
    - 고양이, 개, 자동차 등 1,000가지 클래스를 가진 100만 개의 대형 데이터셋으로 신경망을 먼저 학습시킵니다.
        
    - 이 과정에서 앞단 레이어들의 파라미터($W^{[1]}, b^{[1]}, \dots, W^{[4]}, b^{[4]}$)가 이미지의 보편적 특징을 추출하는 법을 배우게 됩니다.
        
2. **미세 조정 (Fine-tuning):**
    
    - 기존 모델에서 **마지막 출력층만 제거**하고, 내가 풀고자 하는 문제(예: 0~9 손글씨 분류 $\rightarrow$ 출력 유닛 10개)에 맞게 **새로운 출력층($W^{[5]}, b^{[5]}$)을 붙입니다.**
        
    - 사전 학습된 파라미터를 초기값(Starting Point)으로 삼아 내 데이터셋으로 재학습시킵니다.
        
---

### 3). 미세 조정(Fine-tuning)을 위한 2가지 옵션

|**구분**|**옵션 1: 마지막 층만 학습 (Train Output Layer Only)**|**옵션 2: 전체 층 학습 (Train All Layers)**|
|---|---|---|
|**작동 방식**|앞단 레이어($W^{[1]} \sim W^{[4]}$)는 **고정(Freeze)**하고, 새롭게 추가한 **마지막 출력층만 학습**|사전 학습된 값으로 전체 레이어를 초기화한 뒤, **모든 레이어의 가중치를 업데이트**|
|**적용 조건**|내가 가진 데이터셋이 **매우 적을 때**|내가 가진 데이터셋이 **어느 정도 존재할 때**|

> 💡 **실무에서의 팁:**
> 
> 직접 대형 데이터셋을 몇 주 동안 Pre-training시킬 필요 없이, 인터넷에 무료로 공개된(Open Source) **Pre-trained 모델을 다운로드**받아 내 문제에 맞게 Fine-tuning만 진행하는 것이 일반적입니다.


---
### 4). 전이 학습이 잘 작동하는 이유 (직관적 이해)

이미지 인식 신경망의 각 레이어는 단계별로 보편적인 특징(Generic Features)을 학습합니다:

- **Layer 1:** 선/에지(Edges) 감지
    
- **Layer 2:** 에지들이 모여 만든 모서리(Corners) 감지
    
- **Layer 3:** 기본적인 곡선 및 단순한 형태(Basic Shapes) 감지
    

고양이, 개, 자동차를 구분하며 배운 '에지, 모서리, 기본 형태를 찾는 지식'은 손글씨 숫자를 구분하거나 다른 이미지 분류 문제를 풀 때도 똑같이 유용하게 쓰입니다.

---

### 5). 전이 학습 적용 시 제약 조건

- **입력 데이터 타입($X$)이 동일해야 함:**
    
    - 이미지 모델 $\rightarrow$ 이미지 데이터 문제에 적용 가능 (O)
        
    - 이미지 모델 $\rightarrow$ 음성/텍스트 데이터 문제에 적용 불가 (X)
        
- **대표적인 전이 학습 사례:**
    
    - **컴퓨터 비전:** ImageNet 기반 pre-trained 모델
        
    - **자연어 처리(NLP):** BERT, GPT 시리즈 등 대규모 언어 모델(LLM)

## 5. Full Cycle of a machine Learning Project
- 실제 현업에서 성공적인 머신러닝 시스템을 구축하기 위한 **프로젝트 4단계 라이프사이클, 추론 서버(Inference Server) 배포 방식, 그리고 MLOps의 개념**


### 1). 머신러닝 프로젝트의 4단계 라이프사이클

실제 머신러닝 시스템을 만드는 것은 단순히 모델을 훈련하는 것 이상의 과정을 거칩니다. 음성 인식(Voice Search) 프로젝트를 예시로 한 전체 과정은 다음과 같습니다.

```
[1. 프로젝트 정의 (Scoping)]
  └─ 풀어야 할 문제와 목표 결정 (예: 모바일 음성 검색)
           │
           ▼
[2. 데이터 수집 (Data Collection)]
  └─ 음성 데이터 및 텍스트 트랜스크립트(라벨) 수집
           │
           ▼
[3. 모델 학습 및 오류 분석 (Model Training & Iteration)] ──┐ (성능 미달 시 반복)
  ├─ 모델 훈련 ──► 오류 분석 ──► 모자란 특수 데이터 수집   │
  └─ 예: 차 안 소음 문제 발견 ──► 데이터 증강(Augmentation) ┘
           │
           ▼ (성능 목표 달성 시)
[4. 시스템 배포 및 모니터링 (Deployment & Maintenance)]
  └─ 서비스 제공 ──► 성능 모니터링 ──► 데이터 변화 감지 시 재학습 및 업데이트
```

1. **프로젝트 정의 (Scoping):** 만들고자 하는 시스템의 범위와 목표를 정의합니다.
    
2. **데이터 수집 (Data Collection):** 입력($X$)과 출력 라벨($Y$) 데이터를 수집합니다.
    
3. **모델 학습 및 오차 분석 (Model Training & Error Analysis):**
    
    - 모델을 학습시키고 오차를 분석하며 지속적으로 개선합니다.
        
    - **반복 루프(Loop):** 차 안 소음 등 특정 환경에서 오차가 크다면, 해당 데이터 증강(Augmentation)이나 추가 수집을 위해 **2단계로 돌아가서 재학습**을 거칩니다.
        
4. **배포 및 유지보수 (Deployment & Maintenance):** 실제 프로덕션 환경에 모델을 배포하고, 성능이 저하되지 않도록 지속적으로 모니터링 및 업데이트합니다.
    

---

### 2). 프로덕션 배포 방식 (Inference Server & API)

학습된 모델을 실제 서비스(예: 모바일 앱)에 연결하는 표준적인 구조는 다음과 같습니다.

- **추론 서버 (Inference Server):** 학습된 모델을 서버에 탑재하여 예측 요청을 처리합니다.
    
- **API 호출 구조:**
    
    1. 사용자(모바일 앱)가 음성 데이터($X$)를 기록하여 API를 통해 추론 서버로 전송합니다.
        
    2. 추론 서버가 모델을 실행하여 예측된 텍스트($\hat{Y}$)를 응답으로 반환합니다.
        
- **소프트웨어 엔지니어링의 필요성:**
    
    - 소수 사용자를 위한 시스템과 수백만 명의 사용자를 감당하는 시스템은 구조가 다릅니다.
        
    - 서버의 안정성, 확장성(Scaling), 연산 비용 최적화, **로그(Logs) 수집** 등을 위한 소프트웨어 엔지니어링이 필수적입니다.
        

---

### 3). 데이터 변화(Data Drift)와 MLOps의 중요성

#### 데이터 변화(Data Drift)란?

- 트레이닝 시점에는 없던 **새로운 유명인, 정치인 이름, 새로운 유행어** 등이 등장하면 사용자의 검색 패턴이 바뀌어 모델의 정확도가 떨어집니다.
    
- 입력 데이터($X$)와 예측 결과($\hat{Y}$)를 지속적으로 **로그(Log)로 남기고 모니터링**해야만 성능 저하 시점을 파악하고 모델을 재학습(Retrain)시킬 수 있습니다.
    

#### MLOps (Machine Learning Operations)

- 머신러닝 모델을 시스템화하여 **체계적으로 구축, 배포, 모니터링, 유지보수**하는 실무 분야입니다.
    
- 모델이 대규모 사용자 환경에서도 **신뢰할 수 있고(Reliable), 잘 확장되며(Scalable), 적절히 업데이트**되도록 만드는 것이 MLOps의 핵심 목적입니다.
    
---

### 4). 결론 
- 모델 학습은 머신러닝 시스템의 핵심이지만, 실제 서비스로 완성하기 위해서는 **엔지니어링, MLOps, 데이터 모니터링**이 수반되는 전체 라이프사이클을 이해해야 합니다.

## 6. Fairness, Bias, and ethics
- 머신러닝이 수많은 사람들의 삶에 직접적인 영향을 미치는 시대인 만큼, **모델 배포 시 발생할 수 있는 윤리적 문제와 이를 방지하기 위한 실천 가이드라인**
### 1). 머신러닝 편향(Bias) 및 악용의 대표적 사례

과거 잘못 설계되거나 제대로 검증되지 않은 채 배포되어 사회적 문제를 일으킨 대표적인 예시들입니다.

#### (1) 편향 및 차별 문제 (Fairness & Discrimination)

- **채용 도구의 성차별:** 과거 특정 기업의 AI 채용 시스템이 여성 지원자에게 불이익을 주는 편향을 보여 사용이 중단됨.
    
- **얼굴 인식의 인종 편향:** 유색인종의 얼굴을 백인보다 용의자 사진(Mugshot)과 잘못 매칭하는 비율이 현저히 높았음.
    
- **대출 승인 불평등:** 특정 인종이나 계층, 지역 소그룹에 불리하게 대출 승인을 거절하는 편향 발생.
    
- **고정관념 강화:** 검색 결과나 추천 시스템이 성별·인종에 대한 부작용(고정관념)을 지속적으로 학습하고 노출함.
    

#### (2) 기술의 악용 사례 (Adverse / Unethical Use Cases)

- **딥페이크(Deepfake):** 동의 없는 위조 영상 생성 및 유포.
    
- **자극적 콘텐츠 확산:** 유저 참여도(Engagement) 최적화 알고리즘이 혐오 발언이나 자극적 뉴스 확산을 유도.
    
- **자동화된 봇(Bot):** 상업적 가짜 리뷰 작성 또는 정치적 선동 목적의 콘텐츠 자동 생성.
    
- **스팸 및 금융 사기:** 스패머와 사기꾼들이 AI 기술을 악용하여 범죄 효율을 높임.
    
---

### 2). 윤리적 AI 시스템 구축을 위한 4가지 가이드라인

앤드류 응 교수님은 윤리에는 단순한 '5가지 체크리스트' 같은 정답이 없다고 말하며, 대신 실무에서 지켜야 할 **4가지 핵심 절차**를 제안합니다.

```
[1. 다양한 팀 구성] ──► [2. 업계 표준 조사] ──► [3. 배포 전 사전 감사] ──► [4. 완화 계획 및 모니터링]
 (다각도 위험 발굴)      (가이드라인 준수)       (그룹별 편향 측정)        (롤백 및 사고 대응)
```

1. **다양성을 갖춘 팀 구성 (Diverse Team Brainstorming)**
    
    - 성별, 인종, 문화적 배경이 다양한 팀원들과 함께 **"무엇이 잘못될 수 있는가?"** 특히 "취약 계층에게 어떤 피해가 갈 수 있는가?"를 사전에 브레인스토밍합니다.
        
2. **업계 표준 및 가이드라인 조사 (Literature Search & Standards)**
    
    - 금융, 의료 등 자신이 속한 산업 분야에서 정립되고 있는 공정성·윤리 표준 가이드라인을 확인하고 준수합니다.
        
3. **배포 전 시스템 사전 감사 (Audit System Prior to Deployment)**
    
    - 모델 학습 완료 후 배포 직전 단계에서, 특정 그룹(성별, 인종, 연령 등)에 대해 성능 차이(편향)가 존재하는지 수치적으로 측정하고 감사(Audit)를 진행합니다.
        
4. **완화 계획 수립 및 지속적 모니터링 (Mitigation Plan & Continuous Monitoring)**
    
    - 문제가 발생했을 때 즉시 **이전 안전 버전으로 돌아가는 롤백(Rollback) 수단** 등 완화 계획을 미리 마련합니다.
        
    - 자율주행차 사고 대응 플랜처럼 배포 후에도 지속적으로 모니터링하여 위험을 예방합니다.

# < Skewed Datasets > 

## 1. Error Metrics for skewed Datasets
- 불균형 데이터셋을 위한 에러 평가 지표
- 양성(Positive)과 음성(Negative) 클래스의 비율이 50:50에서 크게 벗어난 **불균형 데이터셋(Skewed Dataset)** 환경에서 기존의 정확도(Accuracy) 지표가 가지는 한계점과 이를 극복하기 위한 **정밀도(Precision) 및 재현율(Recall)** 개념을 상세히 정리

### 1). 정확도(Accuracy) 지표의 한계

희귀 질환(전체 환자의 0.5%만 양성) 진단 모델을 개발하는 상황을 가정해 보겠습니다.

- **모델 A (복잡한 ML 모델):** 테스트 세트 오차율 1% (**정확도 99%**)
    
- **모델 B (무조건 0/음성만 출력하는 멍청한 알고리즘):** 테스트 세트 오차율 0.5% (**정확도 99.5%**)
    

단순 오차율/정확도만 비교하면 아무런 분류도 하지 않고 무조건 `y = 0`만 출력하는 모델 B가 더 뛰어난 것처럼 보입니다. 하지만 모델 B는 실제 의료 진단 도구로서 전혀 가치가 없습니다.

> 💡 **핵심 요약:** 불균형 데이터셋에서는 오차율(Error Rate)이나 정확도(Accuracy)만으로 모델의 성능을 판단할 수 없습니다.

---
### 2). 혼동 행렬 (Confusion Matrix)

희귀 클래스($y=1$) 분류 성능을 제대로 평가하기 위해 2x2 형태의 혼동 행렬(Confusion Matrix)을 작성합니다.



|                            | 실제 양성 (Actual Positive, y=1)                              | 실제 음성 (Actual Negative, y=0)                              |
| -------------------------- | --------------------------------------------------------- | --------------------------------------------------------- |
| 예측 양성<br>(Predicted $y=1$) | **True Positive (TP)**<br>• 실제 양성을 **양성으로 맞게 예측**         | False Positive (FP)<br>• 실제 음성을 **양성으로 틀리게 예측** _(거짓 양성)_ |
| 예측 음성<br>(Predicted $y=0$) | False Negative (FN)<br>• 실제 양성을 **음성으로 틀리게 예측** _(거짓 음성)_ | True Negative (TN)<br>• 실제 음성을 **음성으로 맞게 예측**             |

---
### 3). 정밀도(Precision)와 재현율(Recall)

불균형 데이터셋에서는 정확도 대신 **정밀도**와 **재현율**을 함께 측정해야 합니다.

#### (1) 정밀도 (Precision)

- **질문:** _"모델이 양성($y=1$)이라고 예측한 환자들 중, 실제로 병을 가진 비율은 얼마인가?"_
    
- **공식:**
    
    $$\text{Precision} = \frac{\text{True Positives (TP)}}{\text{Predicted Positives (TP + FP)}}$$
    
- **의미:** 모델이 양성이라고 진단했을 때의 **신뢰도/정확성**을 의미합니다.
    

#### (2) 재현율 (Recall)

- **질문:** _"실제 병을 가진 전체 환자들 중, 모델이 제대로 찾아낸(진단한) 비율은 얼마인가?"_
    
- **공식:**
    
    $$\text{Recall} = \frac{\text{True Positives (TP)}}{\text{Actual Positives (TP + FN)}}$$
    
- **의미:** Actual Positive 중 실제 양성을 얼마나 잘 감지/포착(Recall)해냈는지를 의미합니다.
    
---
### 4). 정밀도와 재현율이 멍청한 알고리즘을 잡아내는 방법

만약 무조건 `y = 0`만 출력하는 알고리즘(모델 B)을 적용하면:

- $TP = 0$이 되므로 **Recall = 0**이 됩니다.
    
- 정밀도와 재현율 중 하나라도 0에 가깝다면 해당 알고리즘은 **유용하지 않은 모델**로 즉시 판별할 수 있습니다.
    

> **예시 데이터 기반 계산 (강의 속 예시):**
> 
> - $TP = 15$, $FP = 5$, $FN = 10$, $TN = 70$
>     
> - **Precision:** $15 / (15 + 5) = 0.75$ (**75%**)
>     
> - **Recall:** $15 / (15 + 10) = 0.60$ (**60%**)
>     
---
### 5). 결론
- 희귀 클래스를 탐지할 때는 **Precision과 Recall 두 지표가 모두 적절히 높은지 확인**해야 모델이 실제 유용한지 판단할 수 있습니다.

## 2. Trading off Precision and recall
- **Trading off precision and recall (정밀도와 재현율의 트레이드오프 및 F1 Score)"** 내용


### 1). 정밀도(Precision)와 재현율(Recall)의 트레이드오프

기본적으로 로지스틱 회귀(Logistic Regression)는 $0$과 $1$ 사이의 확률 값($f(x)$)을 출력합니다. 보통은 임계값(Threshold)을 $0.5$로 설정하여 예측합니다.

- $f(x) \ge 0.5 \implies y = 1$
    
- $f(x) < 0.5 \implies y = 0$
    

하지만 문제의 성격에 따라 이 임계값을 조정해야 하며, 이에 따라 **정밀도와 재현율은 서로 반대로 움직이는 트레이드오프(Trade-off) 관계**를 갖습니다.

####  임계값을 높일 때 ($0.5 \to 0.7$ 또는 $0.9$)

- **상황:** 양성($y=1$)이라고 확신할 때만 진단을 내리고 싶은 경우 (예: 치료비가 매우 비싸거나 수술이 위험하여 오진 시 영향이 큰 경우)
    
- **결과:**
    
    - **Precision 상승 ⬆️:** 확실한 경우만 $1$로 예측하므로, $1$이라고 했을 때 맞을 확률이 높아짐.
        
    - **Recall 하락 ⬇️:** $1$로 예측하는 횟수 자체가 줄어들어 실제 환자를 놓칠 확률이 커짐.
        

#### 임계값을 낮출 때 ($0.5 \to 0.3$)

- **상황:** 의심스러우면 일단 양성($y=1$)으로 예측하고 싶은 경우 (예: 질병을 방치했을 때의 위험이 치료 과정보다 훨씬 치명적인 경우)
    
- **결과:**
    
    - **Precision 하락 ⬇️:** 기준이 느슨해져 틀린 예측(False Positive)이 늘어남.
        
    - **Recall 상승 ⬆️:** 실제 환자를 놓치지 않고 최대한 많이 찾아냄.
        
---
### 2). 임계값 선택과 곡선 (Precision-Recall Curve)

임계값을 $0.01$부터 $0.99$까지 변경해 보면서 그래프를 그리면 **Precision-Recall 곡선**이 만들어집니다.

- **수동 선택:** 이 곡선을 통해 모델의 목적(False Positive 비용 vs False Negative 비용)에 맞는 최적의 임계값 포인트를 사람이 직접 선택해야 합니다.
    
- **참고:** 임계값 선택은 교차 검증(Cross-validation)으로 자동화하기 어렵고, 도메인 지식과 비즈니스 상황에 맞춰 결정합니다.
    
---
### 3). F1 Score (조화평균)

여러 모델의 성능을 비교할 때 Precision과 Recall 두 수치가 따로 놀면 어떤 모델이 더 좋은지 한눈에 판단하기 어렵습니다.

> **💡 단순 평균(Arithmetic Mean)의 문제점**
> 
> - 모든 환자에게 무조건 $y=1$을 출력하는 멍청한 모델: **Recall = 1.0 (100%), Precision = 아주 낮음**
>     
> - 단순 평균을 구하면 점수가 높게 나와 좋은 모델처럼 착각할 수 있습니다.
>     

따라서 Precision과 Recall 중 더 낮은 수치에 훨씬 더 가중치를 두는 $F_1$ Score(조화평균, Harmonic Mean)를 사용합니다.

#### $F_1$ Score 계산 공식

$$F_1 = 2 \times \frac{P \times R}{P + R}$$
#### 강의 속 모델 비교 예시

|**모델**|**Precision (P)**|**Recall (R)**|**단순 평균**|**F1​ Score**|**평가**|
|---|---|---|---|---|---|
|**Algorithm 1**|**0.50**|**0.40**|0.45|**0.444**|**최적 (균형 잡힌 성능)**|
|**Algorithm 2**|0.70|0.10|0.40|**0.175**|Recall이 너무 낮아 감점|
|**Algorithm 3**|0.02|1.00|0.51|**0.039**|Precision이 엉망이라 낮은 점수|

- $F_1$ Score를 적용하면 무조건 $y=1$만 출력하는 **Algorithm 3** 같은 쓸모없는 모델을 확실하게 걸러낼 수 있습니다.
    
---
### 4).요약

- **Precision과 Recall은 반비례 관계**이므로 문제의 목적에 맞춰 임계값(Threshold)을 조절해야 합니다.
    
- 두 지표를 하나의 수치로 종합해 최적의 모델을 평가하고 싶을 때는 단순 평균이 아닌 **$F_1$ Score**를 사용합니다.
